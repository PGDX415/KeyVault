# 密钥阁 KeyVault — 开发需求文档

## 〇、App 名称
- **中文名**：密钥阁
- **英文名**：KeyVault
- 命名寓意：像一个存放"钥匙"（账户密码）的阁楼/保险柜，简洁且易于理解产品定位
- 注意：正式提交 App Store 前需在 App Store Connect 中检查名称是否已被占用，
  如冲突可考虑备用名：**锁匣 / KeyBox**、**密钥屋 / KeyCase**

## 〇.一、App 图标要求
- **核心图形元素**：盾牌（Shield），传达安全防护的意象
- **视觉风格**：拟物质感（Skeuomorphic），带有金属质感/立体光影，
  避免纯扁平化图标，突出"保险箱/安全锁"的厚重可信感
  - 可考虑在盾牌上叠加锁孔、锁具或钥匙孔的细节，强化"密码保险箱"的联想
  - 金属质感可通过高光、渐变、细微反光和阴影体现立体感
- **主色调**：紫色系（呼应苹果密码类 App 常见配色，如 Passwords App 的配色体系）
  - 建议使用深紫到浅紫的渐变，或紫色搭配金属银/金色细节提升质感
- **背景**：建议使用纯色或渐变背景（避免复杂背景，保证在主屏幕上清晰可辨识）
- **尺寸与规范**：
  - 需按 Apple 官方 App Icon 规范提供全套尺寸（1024x1024 主图，及各 iPhone/iPad 所需尺寸）
  - 需符合 iOS/iPadOS 图标圆角遮罩规则，设计时预留安全边距，避免关键元素被裁切
  - 需分别适配浅色模式、深色模式图标（iOS 18+ 支持深色图标变体），如时间允许可额外提供
- **风格统一性**：图标视觉语言（紫色+盾牌+金属质感）应与 App 内启动页、
  解锁页等关键界面保持一致，强化品牌辨识度

## 〇.二、启动页（Splash Screen）要求
- **视觉风格与图标保持一致**：延续紫色系 + 盾牌元素的品牌视觉语言，
  让用户从点击图标到进入启动页有连贯的视觉体验
- **背景**：使用纯色或深浅紫渐变背景（可与图标背景色呼应），避免突兀的白屏
- **核心元素**：
  - 居中展示盾牌图形（可用简化版图标，或图标的线条/剪影版本）
  - 品牌名称 **密钥阁 / KeyVault**（图形下方或旁边，字体简洁大方）
- **动效（可选，锦上添花）**：
  - 若技术实现简单，可加入轻微的入场动效，如盾牌淡入 + 轻微缩放，
    或锁具"咔哒"落锁的动画，强化安全感和精致感
  - 若时间/精力有限，纯静态启动页也完全可以接受，优先保证开发进度
- **技术实现**：
  - 使用 iOS 标准的 LaunchScreen 机制（Storyboard 或 SwiftUI 的 Launch Screen 配置），
    确保启动速度快、无明显卡顿
  - 启动页应尽量简单，避免加载远程资源或做复杂计算，保证秒开体验
- **与解锁流程衔接**：
  - 启动页展示后，无缝过渡到 Face ID / Touch ID 或主密码解锁界面，
    过渡应自然流畅，不出现白屏闪烁或明显跳变
- **深色模式适配**：需提供适配深色模式的启动页配色，避免浅色背景在深色模式下显得突兀

## 一、产品概述
一个运行在 iPhone / iPad 上的原生 App，用于安全存储账户名称、密码等敏感信息，
定位类似"密码保险箱"。核心诉求：**安全、简单、可跨设备同步**。

## 二、技术形式
- 原生 iOS / iPadOS App
- 语言：Swift
- UI 框架：SwiftUI
- 最低支持系统版本：iOS 17 / iPadOS 17（可根据开发时的实际情况调整）
- 使用 Xcode 项目结构，方便后续上架 App Store

## 三、核心功能

### 1. 账户记录管理（保险箱主功能）
- 新增 / 编辑 / 删除 一条账户记录
- 每条记录至少包含字段：
  - 名称（如"网站/App 名称"）
  - 账号/用户名
  - 密码
  - 网址（可选）
  - 备注（可选）
  - 创建时间 / 修改时间（自动记录）
- 列表页展示所有记录，支持搜索（按名称/账号模糊搜索）
- 密码字段默认隐藏（显示为圆点），点击可显示/隐藏
- 一键复制账号、密码到剪贴板（复制密码后剪贴板内容建议设置自动过期清空，例如30-60秒后清空，避免残留）

### 2. 安全与解锁
- 使用 **Face ID / Touch ID** 作为日常解锁方式
- 设置一个 **主密码**，作为：
  - 首次设置保险箱时的初始密码
  - 生物识别不可用时的备用解锁方式（如生物识别多次失败、系统未开启生物识别等）
- App 启动或从后台切回前台超过一定时间后，需要重新解锁（自动锁定机制）
- 所有敏感数据在设备本地必须**加密存储**，禁止明文存储
  - 使用 iOS Keychain 存储主密码相关的密钥材料
  - 数据库文件（账户记录）本身也需要加密（如使用 SQLCipher，或通过 Keychain 派生密钥对每条记录/整个数据文件进行 AES 加密后再存储）

### 3. iCloud 自动同步
- 使用 **CloudKit** 实现多设备（iPhone / iPad）之间的自动同步
- 同步应为静默后台同步，用户无需手动操作
- 需处理的场景：
  - 多设备同时编辑的冲突合并（以"最后修改时间较新"为准，或提示用户选择）
  - 网络不可用时先保存本地，恢复网络后自动同步
  - 用户未登录 iCloud 账户时的提示与降级处理（App 仍可离线使用，仅不同步）
- 注意：数据在上传到 iCloud 前应已经过加密，保证即使是 iCloud 侧也不接触明文密码

### 4. 其它建议功能（可选，视开发进度决定是否加入）
- 密码生成器（生成随机高强度密码，可自定义长度/是否包含符号等）
- 密码强度提示
- 收藏/置顶常用账户
- 按分类或标签组织账户（如"社交"、"银行"、"工作"等）
- 深色模式适配

## 四、非功能性需求
- **简单优先**：不做过度设计，优先保证核心记录+加密+同步三件事稳定可靠
- **隐私优先**：不收集任何用户数据，不接入第三方分析/广告 SDK
- 良好的 iPad 适配（如支持分栏布局 NavigationSplitView）
- 支持横竖屏切换

## 五、暂不需要的功能（明确排除，避免过度开发）
- 不需要 Android 或其它平台版本
- 不需要网页版
- 不需要多用户账户体系（App 本身不需要注册登录，只依赖 iCloud 账户做同步）
- 暂不需要手动导出/导入文件备份（后续如有需要可再补充）

## 六、建议开发顺序
1. 项目搭建 + 基础数据模型（Account 记录结构）
2. 本地加密存储（Keychain + 加密数据库/文件）
3. Face ID / Touch ID + 主密码解锁流程
4. 账户记录的增删改查 UI
5. CloudKit 同步接入与测试（多设备同步）
6. 自动锁定、剪贴板自动清空等安全细节打磨
7. 可选功能（密码生成器等）
